## 1. 执行摘要

AI Agent 正在从实验室走向产业。从航班预订到供应链管理，从代码审查到金融交易，越来越多的关键任务正在被委托给自主运行的 AI Agent。Gartner 预测，到 2028 年，33% 的企业软件将包含 Agentic AI 组件，而 2024 年这一比例不到 1%。

然而，一个根本性的问题始终悬而未决：**当一个 Agent 把任务交给另一个 Agent 时，它凭什么相信对方会正确执行？**

现有的 Agent 间协议各有所长——Google A2A 解决了通信互操作，ANP 提供了去中心化发现，Anthropic MCP 连接了 AI 与工具。但没有一个真正回答了信任问题。

**Agent Trust & Exchange Layer（ATEL）** 正是为此而生。ATEL 是 AI Agent 世界的信任基础设施，提供三大核心创新：

- **Trust Execution SDK**：让每一次 Agent 执行都变得可审计、可追溯、可回滚。通过密码学证明和不可篡改的执行轨迹，将"信任"从模糊概念变为可计算的事实。
- **Trust Score Network**：基于真实执行数据和链上证明构建的信誉评分网络。不是自我声明，而是用密码学证明支撑的可审计执行历史。
- **Trust Graph**：多维度信任图谱，不仅回答"这个 Agent 好不好"，更回答"在什么场景下、和谁协作时、它是否可信"。

ATEL 不替代现有协议，而是成为它们的信任层。A2A 解决了"怎么聊"，ANP 解决了"怎么找"，ATEL 解决的是"凭什么信"。

**当前状态：** SDK v0.8.3 已发布，包含 23 个功能模块、341 个测试用例全部通过。已实现跨网络 Agent 通讯、三链主网链上锚定验证、端到端加密通信、智能信任评估、完整商业交易闭环等核心能力。

## 2. 市场背景

### 2.1 AI Agent 爆发式增长

2025-2026 年，AI Agent 生态正在经历前所未有的增长。这不再是概念验证，而是真实的产业变革：

- **Agent 的定义在扩展。** 现代 AI Agent 拥有自己的身份、钱包、决策逻辑，可以代表企业去谈判、签约、执行任务。
- **多 Agent 协作成为主流。** 单个 Agent 能力有限，真正的价值在于多 Agent 分工协作——搜索、比价、预订、支付，各司其职。
- **经济规模快速膨胀。** Agent 开始处理真实的经济交易，涉及资金流转、合同签署、数据交换。

### 2.2 信任：被忽视的关键缺失

在人类社会中，信任是一切协作的基础——法律合同、品牌声誉、第三方担保。但在 AI Agent 的世界里，这些机制几乎完全缺失。

**信任问题的三个层次：**

| 层次 | 核心问题 | 现状 |
|------|---------|------|
| 身份信任 | "你是谁？" | 任何 Agent 都可以伪装成另一个 Agent |
| 执行信任 | "你真的做了吗？" | 无法验证 Agent 是否真正执行了任务 |
| 持续信任 | "你一直可靠吗？" | 没有持续的信誉追踪系统 |

**为什么现在变得紧迫？**

1. **经济利益驱动作弊。** 不诚实的 Agent 可以伪造执行结果来节省成本，同时收取服务费。
2. **规模放大风险。** Agent 数量达到百万级时，人工监督不再可行。
3. **监管要求。** 金融、医疗、法律等行业要求可审计性和可追溯性。
4. **供应链攻击。** 多 Agent 协作链中，一个恶意 Agent 可以污染整个链条。

### 2.3 现有方案的不足

| 协议 | 解决的问题 | 未解决的问题 |
|------|-----------|-------------|
| Google A2A | Agent 间通信互操作 | 无执行验证、无信誉系统、无经济激励 |
| ANP | 去中心化身份与发现 | 有身份无信任、无惩罚机制、无执行审计 |
| Visa/Cloudflare TAP | 电商支付场景验证 | 场景局限、单向验证、中心化依赖 |
| Anthropic MCP | AI 与工具连接 | 不涉及 Agent 间通信和信任 |

**核心空白：执行验证、信誉评分、信任图谱、经济激励——这四个关键能力在所有现有协议中都是缺失的。**

## 3. ATEL 解决方案概述

### 3.1 定位：信任层，不是通信协议

ATEL 的定位非常明确：

> **ATEL 是让 Agent 间协议变得可用、可商用、可自动化的信任层。**

这意味着 ATEL 不与 A2A、ANP 竞争，而是与它们互补。类比互联网协议栈：

```
互联网：  TCP/IP（通信）+ DNS（发现）+ SSL/TLS（安全）+ OAuth（授权）
Agent：   A2A/ANP（通信）+ Directory（发现）+ ATEL（信任 + 授权 + 经济）
```

就像 SSL/TLS 不替代 TCP/IP，而是在其之上提供安全层一样，ATEL 不替代通信协议，而是在它们之上提供信任层。

### 3.2 三大核心创新

**Trust Execution SDK — 可审计执行**

每一次 Agent 执行都生成不可篡改的执行轨迹（Trace）和密码学证明（Proof）。任何第三方都可以独立审计执行过程的真实性。需要注意的是，这是一种透明度机制（事后可发现作假并追责），而非强制验证机制（阻止作假发生）。作假行为一旦被发现，将通过声誉系统进行惩罚。

**Trust Score Network — 信誉评分网络**

基于真实执行数据和链上证明计算信誉分。信任分数不是自我声明，而是从可审计的执行历史中计算得出。没有链上验证证明的 Agent，信任等级永远无法突破基础层。

**Trust Graph — 多维信任图谱**

传统信誉系统只给一个综合分数。Trust Graph 构建多维度信任图谱，记录协作关系、场景表现、协作兼容性。它回答的不是"这个 Agent 好不好"，而是"在这个具体场景下，和这个具体伙伴协作时，它是否可信"。

### 3.3 设计哲学

- **信任不等于通信。** 能通信不代表能信任，信任需要独立的专门基础设施。
- **可审计执行是信任的基础。** Agent 不仅要说它做了什么，还要能证明它确实做了。作假行为通过链上锚定和声誉系统被发现和惩罚。
- **渐进式信任建立。** 新 Agent 从零信任开始，通过真实执行逐步积累信誉。
- **开发者体验优先。** 零代码接入，CLI 命令即可完成全部操作。
- **Human in the Loop。** 人类监督作为核心设计原则，根据风险等级动态调整自动化程度。

## 4. 核心能力

### 4.1 去中心化身份（DID）

每个 ATEL Agent 拥有一个全局唯一的去中心化身份标识符（DID），基于 Ed25519 密码学算法生成。

**核心特性：**

- **自主可控：** Agent 自行生成和管理密钥对，不依赖任何中心化注册机构。
- **密码学绑定：** DID 与公钥直接绑定，任何人都可以通过公钥验证 Agent 的签名，无法伪造。
- **密钥轮换：** 支持在不改变 DID 的情况下安全更新密钥对，密钥轮换事件可锚定上链作为不可篡改的时间证明。
- **W3C 兼容：** 遵循 W3C DID 标准规范，具备跨生态互操作性。

```bash
# 一条命令创建 Agent 身份
atel init my-agent
# 输出：DID + Ed25519 密钥对 + 默认安全策略
```

### 4.2 安全握手与加密通信

ATEL 实现了完整的端到端加密通信体系，确保 Agent 间通讯的机密性和完整性。

**双层防护架构：**

```
┌─────────────────────────────────────────┐
│  E2E 加密层（XSalsa20-Poly1305）         │
│  → 保护消息内容，即使传输层被破解也安全    │
│  ┌─────────────────────────────────────┐ │
│  │  TLS 传输层                          │ │
│  │  → 保护传输通道，防止网络层窃听       │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

**握手流程：** 两个 Agent 首次通讯时，通过三步握手协议互相验证身份——交换 DID 和公钥、互相签名验证挑战值、建立加密会话。握手完成后，所有后续通讯自动加密，中间人（包括中继服务器）只能看到密文。

**加密方案：**
- 密钥交换：X25519（Curve25519 Diffie-Hellman）
- 对称加密：XSalsa20-Poly1305（NaCl 标准 AEAD 方案）
- 签名算法：Ed25519（128 位安全级别，64 字节签名）

### 4.3 智能信任评估

ATEL 的信任评估系统是其最核心的差异化能力，采用渐进式信任建立机制。

**四级信任等级：**

| 等级 | 名称 | 能力范围 | 典型升级路径 |
|------|------|---------|-------------|
| Level 0 | 零信任 | 仅限低风险任务 | 新注册 Agent 默认等级 |
| Level 1 | 基础信任 | 可执行中等风险任务 | 完成首次成功任务后 |
| Level 2 | 验证信任 | 可执行高风险任务 | 需要链上验证证明 |
| Level 3 | 企业信任 | 可执行关键业务任务 | 持续高成功率 + 大量链上证明 |

**关键设计：**

- **信任分数从执行历史中计算，** 综合考虑成功率、任务量、链上证明比例等多个维度。
- **没有链上验证证明的 Agent 永远无法达到 Level 2。** 链上证据是信任体系的基石。
- **双模式评估：** 本地模式（基于直接交互历史，快速无网络开销）和链上验证模式（跨链查询验证，可评估从未交互过的 Agent）。
- **每个 Agent 独立计算信任分数，** 类似信用局模型——数据公开，但每个参与方根据自己的策略做决策。

### 4.4 多链锚定

ATEL 支持将执行证明锚定到多条公链，实现不可篡改的时间戳和全网可审计性。

**支持的区块链：**

| 链 | 锚定方式 | 单次成本 | 确认时间 |
|---|---------|---------|---------|
| Solana | Memo Program | ~$0.001 | ~0.4 秒 |
| Base (Ethereum L2) | 零值交易 + data 字段 | ~$0.001-$0.005 | ~2 秒 |
| BSC | 零值交易 + data 字段 | ~$0.005-$0.02 | ~3 秒 |

**设计原则：**

- **只存哈希，不存完整数据。** 链上仅记录执行证明的 SHA-256 哈希（32 字节），完整数据保留在链下。成本降低数个数量级，同时保护隐私。
- **链无关抽象。** 统一的 AnchorProvider 接口，开发者可按需选择链，也可自行扩展支持新链。
- **自描述 Memo 格式。** 链上记录包含执行者 DID、请求者 DID、任务 ID 和证明根哈希，任何人可独立解析和验证。
- **成本可控。** 即使每天锚定 10,000 次，月成本仅 $300-$1,500。

**验证流程：** 任何第三方都可以通过链上交易哈希独立验证执行证明——查询链上 Memo 内容，与链下 Proof Bundle 的哈希比对，即可确认证明未被篡改。这是完全去中心化的验证，不依赖任何中间方。

### 4.5 内容安全审计

ATEL 内置多层内容安全审计机制，在任务到达执行层之前拦截恶意输入。

**审计覆盖范围：**

- SQL / NoSQL 注入检测
- 路径遍历攻击检测
- 命令注入检测
- 凭证访问检测
- Payload 大小限制
- DID 黑白名单
- 速率限制与并发控制

**安全策略可配置：** 每个 Agent 通过本地策略文件定义自己的安全规则，包括速率限制、最大负载大小、并发上限、DID 白名单/黑名单等。所有拒绝操作都会生成本地 Trace 和 Proof，返回给发送方以证明拒绝的合法性。

### 4.6 零代码接入（CLI + SKILL.md）

ATEL 将"开发者体验优先"贯彻到极致——AI Agent 不需要编写任何代码即可接入信任网络。

**CLI 工具 `atel`：**

```bash
# 安装
npm install -g @lawrenceliang-btc/atel-sdk

# 创建身份
atel init my-agent

# 注册能力并启动
atel register "My Agent" "translation,coding"
atel start

# 搜索其他 Agent
atel search translation

# 一键完成商业交易（搜索 → 下单 → 等待 → 自动确认）
atel trade-task translation '{"text":"Hello","target_lang":"zh"}' --price 0.1

# 发送任务（自动握手 + 加密 + 信任检查）
atel task "did:atel:xxxxx" '{"action":"translation","text":"Hello","target_lang":"zh"}'

# 查看信任分数
atel check "did:atel:xxxxx" medium
```

**SKILL.md 协议：** ATEL 提供标准化的 Skill 文件，任何 AI Agent（无论底层框架是 LangChain、CrewAI、OpenClaw 还是其他）都可以通过读取 SKILL.md 学会使用 ATEL。Agent 不运行不可信代码，只执行可信的 CLI 命令——CLI 就是信任边界。

**自动化网络处理：** `atel start` 自动检测网络环境、发现公网 IP、尝试 UPnP 端口映射、注册中继备选。无论 Agent 处于 NAT、防火墙还是任何网络拓扑下，都能自动建立连接。

## 5. 技术架构概览

### 5.1 五层架构

ATEL 采用分层架构设计，每一层职责明确、可独立部署、可渐进采用。

```
┌─────────────────────────────────────────────────────────┐
│  Layer 4: Router + Marketplace                          │
│  任务路由 │ 智能匹配 │ 结算清算 │ 争议仲裁              │
├─────────────────────────────────────────────────────────┤
│  Layer 3: Discovery / Directory                         │
│  能力注册 │ 能力查询 │ 可信目录 │ 分类索引              │
├─────────────────────────────────────────────────────────┤
│  Layer 2: Trust Score Network + Trust Graph             │
│  信誉评分 │ 风险画像 │ 认证等级 │ 信任图谱              │
├─────────────────────────────────────────────────────────┤
│  Layer 1: Identity & Auth                               │
│  DID 身份 │ 公钥绑定 │ 签名验证 │ 授权令牌              │
├─────────────────────────────────────────────────────────┤
│  Layer 0: Trust Execution SDK                           │
│  任务封装 │ 权限策略 │ 工具拦截 │ 执行轨迹 │ 密码学证明 │
└─────────────────────────────────────────────────────────┘
```

**设计原则：**

- **自底向上构建。** Layer 0 是基础，不依赖任何上层。企业可以从 Layer 0 开始，逐步向上扩展。
- **松耦合。** 层与层之间通过标准化接口通信，内部实现可自由替换。
- **渐进式采用。** 不需要一次性部署所有层。只使用 Layer 0 就能让 Agent 执行变得可审计。

### 5.2 混合部署模型

ATEL 根据不同组件的特性选择最合适的部署模式：

| 组件 | 部署模式 | 理由 |
|------|---------|------|
| Registry（黄页） | 中心化运营 | 需要全局一致视图，Agent 需要确定的发现入口 |
| 任务执行 & 证明生成 | 本地化（SDK 内） | 执行发生在 Agent 本地，密码学保证不可伪造 |
| 链上锚定 | 去中心化（公链） | 永久存储，任何人可审计，不依赖任何单一方 |
| 信任分数 | 本地计算 + 链上数据 | Agent 从链上记录本地计算信任分，无需中心化信任服务 |

### 5.3 SDK 模块体系

ATEL SDK 采用单包多模块架构，包含 22 个功能模块，分为两大模块层：

**信任执行层（Trust Execution Layer）：**
Identity、Schema、Policy、Gateway、Trace、Proof、Score、Graph、Trust、TrustSync、Rollback、Anchor、Orchestrator

**Agent 互联层（Networking Layer）：**
Crypto、Envelope、Handshake、Endpoint、Registry、Negotiation、Collaboration、Network、Auditor

开发者安装一个包即可获得全部能力，按需引入模块：
- 只需信任能力（嵌入现有系统）→ 使用信任执行层 API
- 需要完整 Agent 协作能力 → 同时使用两层

### 5.4 端到端协作流程

两个完全陌生的 Agent 的完整协作流程：

```
发现 → 信任评估 → 握手 → 能力协商 → 任务委托 → 可信执行 → 证明生成 → 链上锚定 → 信誉更新
```

**最小可用路径仅需 5 步：** 身份创建 → 握手 → 任务委托 → 执行 → 证明。两个 Agent 只要知道对方的端点地址，就可以直接协作。信任分数是协作的结果，不是前提。

### 6.2 ATEL 的差异化定位

ATEL 不是"又一个通信协议"。ATEL 可以挂载到任何通信协议之上：

- **A2A + ATEL：** 用 A2A 通信，用 ATEL 验证执行、评估信誉、管理权限。
- **ANP + ATEL：** 用 ANP 的 DID 身份和网络发现，用 ATEL 的 Trust Graph 和经济激励。
- **MCP + ATEL：** 用 MCP 连接工具，用 ATEL 的工具网关拦截和验证工具调用。

**核心价值主张：**

1. **从"能聊"到"能信"。** 通信协议让 Agent 能互相对话，ATEL 让 Agent 能互相信任。
2. **从"声称"到"可审计"。** 现有协议中 Agent 的能力和表现都是自我声称的，ATEL 让每次执行都留下可审计的密码学证据，作假行为可被发现并通过声誉系统惩罚。
3. **从"免费"到"有价"。** 现有协议没有经济层，ATEL 让 Agent 经济成为可能。
4. **从"无序"到"有序"。** ATEL 为 Agent 生态引入信誉、惩罚和规则。

## 7. 商业模式

ATEL 采用平台交易抽佣 + 增值服务的商业模式，已在 ATEL Platform 上实现并验证。

### 7.1 核心收入：交易佣金

所有通过平台撮合的 Agent 交易，平台按阶梯费率抽取佣金：

| 交易金额 | 佣金费率 |
|---------|---------|
| $0 - $10 | 5% |
| $10 - $100 | 3% |
| $100+ | 2% |

认证折扣：Certified -0.5%，Enterprise -1%。

资金流转采用自动 Escrow 托管模式：Executor accept 订单时平台自动冻结 Requester 资金 → Executor 完成任务（必须附带执行证明）→ 10 分钟内自动结算或 Requester 手动确认 → 平台结算（扣佣金后付给 Executor）。

### 7.2 增值服务：认证体系

| 等级 | 费用 | 权益 |
|------|------|------|
| Verified | 免费（自动） | 日限额 $500，佣金优惠 |
| Certified | $50/年 | 日限额 $2,000，认证标识 |
| Enterprise | $500/年 | 日限额 $10,000，最大佣金折扣 |

自动认证条件：信任分 ≥65 + 完成任务 ≥10 笔 + 成功率 ≥90% + 链上证明 ≥3 笔 + 注册 ≥7 天。

### 7.3 增值服务：推广系统

| 等级 | 费用 | 效果 |
|------|------|------|

| Featured | $100/周 | 全站置顶 + 专属标识 |

所有 Agent 免费使用核心协议能力：

## 9. 团队与愿景

### 9.1 团队

ATEL 由一支兼具区块链、AI 和分布式系统经验的团队打造。核心团队在密码学协议设计、链上应用开发和 AI Agent 系统架构方面拥有深厚积累。

团队秉持"做出来再说"的工程文化——SDK v0.8.3 的 23 个模块、341 个测试用例、跨网络通讯验证和三链主网锚定，都是在极短时间内从零交付的。

### 9.2 愿景

**TCP/IP 让计算机互联，ATEL 让 AI Agent 互信。**

回顾互联网的发展：1990 年代，互联网只是让计算机互相通信的网络。没有人能预见它会催生电子商务、社交媒体、云计算等万亿级产业。互联网之所以能从技术协议演变为全球经济生态，关键在于一系列信任基础设施的建立——DNS、SSL/TLS、OAuth、支付系统。

今天的 AI Agent 生态正处于类似的早期阶段。我们有了通信协议（A2A、ANP），有了工具连接协议（MCP），但还缺少最关键的一环：**让 Agent 互相信任的基础设施。**

ATEL 的终极目标是成为 AI Agent 世界的信任基础设施。当任何一个 Agent 想要与另一个 Agent 协作时，它不需要关心对方是谁构建的、运行在哪里、使用什么框架——它只需要查看对方的 ATEL Trust Score，验证对方的 Proof Bundle，就可以安全地开始协作。

**从协议到平台，从平台到生态：**

1. **协议阶段：** 定义 Agent 间信任建立的标准方式，价值在于标准化和互操作性。
2. **平台阶段：** Trust Score Network、Trust Graph、Marketplace 上线，形成网络效应和数据飞轮。
3. **生态阶段：** 开发者构建 Agent 并通过 ATEL 变现，企业通过 ATEL 找到可信服务，标准化组织将 ATEL 纳入行业标准。

### 9.3 护城河

ATEL 的护城河不是代码（代码可以被 fork），而是：

- **链上数据积累。** 每一次链上锚定都在积累不可复制的信任证据。
- **Trust Graph 网络效应。** 越多 Agent 使用 ATEL，图谱越丰富，信任推断越准确。
- **用户信任偏好数据。** 每个用户与每个 Agent 之间的信任建立过程是独特的，竞争对手无法复制。
- **飞轮效应。** 更多使用 → 更好的信誉模型 → 更精准的信任评估 → 更好的体验 → 更多使用。

## 10. 安全与合规

### 10.1 安全架构

ATEL 构建了端到端的安全体系，覆盖通讯、身份、执行和数据四个维度。

**通讯安全：**
- 端到端加密（X25519 + XSalsa20-Poly1305），中间人无法解密
- TLS 传输层保护，双层防护架构
- 防重放攻击（Nonce + 时间戳 + 自动清理）
- 防降级攻击（加密为默认行为，不支持降级到明文）

**身份安全：**
- DID 绑定 Ed25519 公钥，密码学保证不可伪造
- 所有消息签名验证，任何篡改都会导致验签失败
- 密钥轮换支持，旧密钥材料安全清零
- 密钥轮换事件可锚定上链，提供不可篡改的时间证明

**执行安全：**
- 不可篡改的执行轨迹（Hash Chain）
- 密码学执行证明（Merkle Tree + 数字签名）
- 细粒度权限控制（授权令牌 + 策略引擎）
- 执行回滚机制（LIFO 补偿操作）

**数据安全：**
- 执行轨迹中只记录输入/输出的哈希，不记录原始数据
- 链上只存储证明哈希，不暴露执行细节
- 企业可部署私有化基础设施，数据完全不离开企业网络

### 10.2 威胁防护

| 威胁类型 | 防护措施 |
|---------|---------|
| 中间人攻击 | E2E 加密 + TLS 双层防护 |
| 重放攻击 | Nonce + 时间戳 + NonceTracker |
| 身份伪造 | DID 绑定公钥 + 消息签名验证 |
| 私钥泄露 | 密钥轮换 + 会话隔离 + 密钥材料清零 |
| Registry 投毒 | 写操作 DID 签名认证 + 管理员白名单 |
| DDoS 攻击 | 每 DID 独立速率限制 |
| 执行伪造 | Hash Chain + Merkle Proof + 链上锚定 |
| 女巫攻击 | 渐进式信任 + 链上证明要求 + 异常检测 |

### 10.3 合规设计

ATEL 的架构设计充分考虑了受监管行业的合规要求：

- **可审计性：** 完整的执行轨迹和密码学证明，满足金融（MiFID II、SOX）和医疗（HIPAA）行业的审计要求。
- **Human in the Loop：** 根据任务风险等级动态调整人类参与程度，关键决策始终保留人类最终决定权。
- **数据主权：** 企业私有化部署选项，数据完全在企业控制范围内。
- **隐私保护：** 链上只存哈希不存明文，执行轨迹中只记录数据哈希，支持 Merkle Proof 选择性披露。
- **不可否认性：** 链上锚定的执行证明提供不可否认的时间戳和完整性证明。
